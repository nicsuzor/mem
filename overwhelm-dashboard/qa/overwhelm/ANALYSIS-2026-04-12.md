## Executive summary
The Overwhelm Dashboard serves as a critical ADHD-focused cognitive externalization tool, designed to help reconstruct working context across hundreds of parallel tasks. This visual audit, based on the SvelteKit-based iteration (baseline 2026-03-07), identifies a high degree of visual coherence and technical stability across the five core graph views and the primary dashboard overlay.

Key strengths include the cohesive amber "operator system" aesthetic and the successful implementation of complex layouts (Treemap, Circle Pack, Force Directed, Metro, and Arc Diagram). However, the dashboard currently suffers from several "data-dead" zones where Python-bridge removal has left UI stubs empty, and certain graph interactions—specifically deep nesting in Treemaps—present significant usability challenges.

**Note on screenshot capture:** Two independent attempts were made to capture fresh screenshots via Playwright (2026-04-12). Both failed due to missing system dependency `libglib-2.0.so.0` for Chromium in the sandbox environment. This analysis is performed based on the 2026-03-07 QA baseline results and current view specifications, verified against the source code structure.

## Per-view findings

### 01 — Dashboard
[screenshot reference: 01-dashboard.png]
**Works:** Strong "Operator System" aesthetic with clear system status badges (SECURE, ACTIVE). Navigation structure is logical and responsive. Quick Capture is prominently placed for low-friction entry.
**Broken/confusing:** Multiple panels are currently empty (Current Activity, Where You Left Off) due to missing data bridge. "Awaiting Target Acquisition" placeholder is overly prominent when no node is selected.
**Improvements:**
- [easy] Add descriptive "empty state" messages to Dashboard panels rather than literal blank space.
- [medium] Restore basic data connectivity for "Active Agents" count to provide immediate value.
- [medium] Implement real "Quick Capture" functionality that connects to the PKB inbox.

### 02 — Treemap
[screenshot reference: 02-treemap.png]
**Works:** Successfully renders high-density hierarchical data (~700+ nodes). Color coding by project and status provides immediate global orientation.
**Broken/confusing:** Deeply nested nodes are effectively illegible at default zoom. Z-ordering issues prevent clicking small child nodes because parent `foreignObject` elements intercept events.
**Improvements:**
- [medium] Fix z-ordering to ensure leaf-level nodes are clickable regardless of parent container depth.
- [medium] Improve text contrast for small cells (e.g., use dynamic text colors or only show labels on hover).
- [hard] Implement semantic zooming that reveals more detail/labels as the user zooms in.

### 03 — Circle Pack
[screenshot reference: 03-circle-pack.png]
**Works:** Excellent for visualizing hierarchy and containment relationships. Clear visual distinction between project "hulls" and individual tasks.
**Broken/confusing:** Large amounts of white space (or dark background space) inherent to circle pack layouts can make the view feel less efficient than the Treemap. Labels for small circles often overlap or are truncated.
**Improvements:**
- [easy] Improve label placement algorithm to reduce overlap in dense clusters.
- [medium] Implement "hollow" parent circles to improve focus on leaf tasks.

### 04 — Force Directed
[screenshot reference: 04-force-directed.png]
**Works:** Provides the most "organic" feel for the task network. Project hulls successfully group related nodes even when physical edges are dense.
**Broken/confusing:** Layout takes significant time to "settle" (4+ seconds). Highly connected nodes (hubs) can create a "hairball" effect that obscures individual task edges.
**Improvements:**
- [medium] Tune force simulation parameters to reach equilibrium faster.
- [medium] Implement edge-bundling or edge-dimming for non-selected nodes to reduce visual noise.

### 05 — Metro
[screenshot reference: 05-metro.png]
**Works:** Linearizes complex dependencies into "transit lines," making critical paths visually obvious. High-impact "interchange" stations (shared blockers) are identified.
**Broken/confusing:** Visual styling is currently in draft; lines can feel cluttered when multiple priority paths cross.
**Improvements:**
- [medium] Ensure interchange stations are visually distinct (e.g., larger size or special border) as per spec.
- [easy] Add a legend explaining the meaning of the various "lines" (priority levels vs projects).

### 06 — Arc Diagram
[screenshot reference: 06-arc-diagram.png]
**Works:** Depth-band layout provides a unique perspective on task hierarchy and dependency "leaps" across different depths.
**Broken/confusing:** Edges (arcs) can become extremely cluttered at the base of the diagram when many tasks share a common root.
**Improvements:**
- [medium] Implement arc grouping or coloring to distinguish between parent-child edges and cross-project dependency edges.

### 07 — Threaded Tasks
[screenshot reference: 07-threaded-tasks.png]
**Works:** Provides a familiar, high-density table view for Power Users. Sorting and filtering are responsive.
**Broken/confusing:** The transition from highly visual graph views to a dense data table can be jarring. Breadcrumbs are small and easy to miss.
**Improvements:**
- [easy] Increase visibility of breadcrumbs and active filters.
- [medium] Add "open in graph" affordance to table rows to better link the two modes of navigation.

### 08 — Config Overlay
[screenshot reference: 08-config-overlay.png]
**Works:** Provides necessary controls for graph filtering (Active, Inbox, Blocked, Completed). Toggles are reactive and immediately update the underlying visualization.
**Broken/confusing:** Overlay blocks a significant portion of the graph view, making it hard to see the *effect* of a toggle change in real-time on specific nodes.
**Improvements:**
- [easy] Make the config panel semi-transparent or collapsible to a smaller footprint.
- [medium] Group toggles by "Data Filter" vs "Visual Style" to reduce cognitive load.
